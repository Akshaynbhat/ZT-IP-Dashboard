The proposed framework employs a multi-layered methodology that translates raw security events into actionable access control decisions. This section details the dataset configuration, the feature engineering pipeline, the mathematical formulation of the hybrid machine learning models, the Trust Score calculation, and the explainability engine.

## A. Dataset and Class Balance

The empirical validation of the proposed system is conducted using the CERT Insider Threat Dataset r4.2, which serves as the standard benchmark for evaluating insider threat detection algorithms. CERT r4.2 contains synthetic log files simulating the digital footprint of employees in a hypothetical organization over an 18-month period. The dataset consists of logs for logon/logoff events, device connections (USB insertion/removal), email metadata, web browsing history, and file access records.

To simulate realistic deployment conditions, this work filters the raw events to isolate logs representing software developers and engineers. A subset of 370 users is extracted, generating a total of 1,667,072 individual event logs. Malicious insider scenarios (such as unauthorized code duplication, credential sharing, and data exfiltration) are represented as a minute minority of these logs. This creates an extreme class imbalance, where malicious events constitute less than 1.27% of the total daily user sessions.

A key methodological decision is the use of a *user-level split* rather than a standard *row-level split* during dataset partitioning. In security applications, randomly splitting individual rows of behavioral logs into training and testing sets leads to severe data leakage. Because behavioral patterns are highly user-specific, a model trained on a subset of a user's daily actions can easily classify other actions of the same user in the test set. This artificially inflates performance metrics and fails to evaluate how the system performs on unseen users. To prevent this, the dataset is split such that 80% of the users are allocated for model training, and the remaining 20% of users are reserved for testing. This design forces the machine learning pipeline to generalize and detect anomalous behaviors in entirely unseen developer accounts.

## B. Feature Engineering

A rolling 24-hour window is used to aggregate raw event logs into daily behavioral vectors for each user. A total of 12 features are engineered to capture patterns indicative of insider threat vectors, such as data hoarding, off-hours exploration, and credential compromise. Table I lists these features along with their categories and the specific insider behaviors they capture.

### Table I: Engineered Features for Insider Threat Detection

| Feature Name | Category | What Insider Behavior it Captures |
| :--- | :--- | :--- |
| `daily_file_reads` | File Access | Detects unusual read volume on source code and repository directories, indicating IP harvesting. |
| `daily_file_writes` | File Access | Measures frequency of file modifications, capturing potential unauthorized code alterations or sabotage. |
| `usb_connect_count` | Device Usage | Tracks connection of external storage devices, indicating potential data exfiltration channels. |
| `off_hours_activity_ratio` | Working Hours | Ratio of events occurring outside core hours (8:00 PM to 6:00 AM), indicating stealth activities. |
| `daily_email_attachments` | Communication | Count of files attached to outgoing external emails, indicating potential source code exfiltration. |
| `external_domain_visits` | Web Browsing | Frequency of visits to file sharing, personal cloud, or unapproved domains. |
| `repo_clones_count` | Repo Operations | Tracks git clone operations, highlighting mass local downloading of company codebases. |
| `failed_logins` | Authentication | Counts consecutive authentication failures, pointing to potential credential theft or brute-forcing. |
| `session_duration` | Session Telemetry | Total active session time in a 24-hour period, identifying prolonged or unauthorized active sessions. |
| `ip_diversity_count` | Session Telemetry | Number of unique source IP addresses per day, capturing potential session hijacking or account sharing. |
| `sensitive_file_deletes` | File Access | Counts deletions of files in restricted source paths, capturing potential codebase sabotage. |

## C. The Two-Stage Hybrid ML Pipeline

The detection pipeline consists of a two-stage hybrid machine learning model designed to handle extreme class imbalance and identify novel anomalous behaviors.

### 1) Stage 1 — Isolation Forest (IF):
The first stage employs an unsupervised Isolation Forest (IF) model. Isolation Forest is highly effective for identifying anomalies in high-dimensional spaces without requiring labeled data. The algorithm isolates observations by randomly selecting a feature and then randomly selecting a split value between the maximum and minimum values. Observations that require fewer splits to isolate are classified as anomalies. 

In this work, the Isolation Forest is configured with a contamination parameter of "auto" and 200 estimators. The model processes the first 18 behavioral features in Table I. The output is a continuous anomaly score $A(x) \in [0, 1]$, where:
$$A(x) = 2^{-\frac{E(h(x))}{c(n)}}$$
Here, $h(x)$ is the path length of observation $x$ in a set of isolation trees, $E(h(x))$ is the average path length, and $c(n)$ is the average path length of an unsuccessful search in a Binary Search Tree. An anomaly score close to 1 represents an extreme outlier.

### 2) Stage 2 — Random Forest (RF):
The second stage utilizes a supervised Random Forest (RF) classifier. To handle the extreme class imbalance in the target labels, the RF model is trained with the hyperparameter `class_weight='balanced'`. This dynamically adjusts weights inversely proportional to class frequencies in the input data:
$$W_j = \frac{N}{C \times n_j}$$
where $W_j$ is the weight of class $j$, $N$ is the total number of samples, $C$ is the number of classes, and $n_j$ is the number of samples in class $j$. The supervised model is configured with `n_estimators=300`, `max_depth=12`, and `min_samples_split=5`.

### 3) Hybrid Feature Augmentation:
The core novelty of this architecture is the hybrid integration of the two models. Instead of running them in parallel, the continuous anomaly score $A(x)$ generated by the unsupervised Isolation Forest is injected directly into the feature vector of the supervised Random Forest as the nineteenth feature (`if_anomaly_score`). 

This hybrid design combines the benefits of unsupervised anomaly detection (which generalizes to unknown attack vectors) with supervised classification (which learns specific historical signatures of insider theft). The Random Forest leverages the global outlier context provided by $A(x)$ to establish a more refined decision boundary, thereby reducing false-positive rates on normal developers working anomalous hours while remaining highly sensitive to low-footprint attacks.

## D. Trust Score Formulation

Dynamic access control decisions are governed by a formalized Trust Score ($T$), calculated at the Policy Decision Point (PDP). The Trust Score represents a mathematical synthesis of behavioral anomaly, supervised risk probability, and identity verification assurance. The score is calculated as follows:
$$T = 100 \times \left[ w_1 \times (1 - A) + w_2 \times (1 - P_r) + w_3 \times C_{id} \right]$$
Where:
- $A \in [0, 1]$ is the unsupervised anomaly score from the Isolation Forest.
- $P_r \in [0, 1]$ is the supervised risk probability of the session being malicious, output by the Random Forest classifier.
- $C_{id} \in [0, 1]$ is the identity confidence score generated by the OIDC provider (Keycloak), calculated based on credentials used (e.g., password vs. MFA), session age, and device compliance status.
- $w_1, w_2, w_3$ are weights representing the relative importance of each dimension, satisfying the constraint:
$$w_1 + w_2 + w_3 = 1.0$$

In this framework, the weights are set to $w_1 = 0.4$, $w_2 = 0.4$, and $w_3 = 0.2$. This structure places primary weight ($80\%$) on behavioral signals, while reserving $20\%$ for identity security, ensuring that compromises in authentication credentials trigger immediate decreases in the Trust Score. The Trust Score is scaled from 0 (complete untrustworthiness) to 100 (maximum trustworthiness).

## E. SHAP-Based Explainability

To provide transparency at the access decision point, the framework integrates SHAP (SHapley Additive exPlanations). We utilize `TreeExplainer`, which is mathematically optimized to calculate exact Shapley values for tree-based ensemble models. For each evaluated session, the SHAP value $\phi_i$ is calculated for each of the 19 features, representing the additive contribution of feature $i$ in shifting the model's prediction from the base value $E[f(x)]$ (the expected prediction over the training set) to the final output $f(x)$:
$$f(x) = E[f(x)] + \sum_{i=1}^{M} \phi_i$$

At the Policy Decision Point, when a session's Trust Score falls below a specific threshold, the SHAP engine extracts the Shapley values. The engine isolates features where $\phi_i > 0$, indicating a positive contribution toward the risk probability (i.e., driving the decision toward a malicious classification). These features are sorted in descending order of their SHAP values, and the top three features are extracted. The PDP maps these features to human-readable reason codes (e.g., "Anomalous off-hours repository write volume") that are surfaced to security administrators in real-time. This provides clear justification for automated access restrictions, resolving the "black box" limitation of traditional machine learning security tools.
