The security of software development environments represents a primary concern for modern enterprises, where source code repositories and proprietary intellectual property (IP) constitute the core assets of the organization. Traditional perimeter security mechanisms, which rely on the "trust but verify" paradigm, focus on securing the network boundary through firewalls, Virtual Private Networks (VPNs), and initial multi-factor authentication (MFA). However, once an actor penetrates this perimeter, they are granted broad lateral access to sensitive codebases. This approach fails to address the threat posed by authenticated insiders—legitimate employees, contractors, or compromised accounts—who leverage authorized credentials to exfiltrate proprietary source code or sabotage software repositories. The detection of such malicious insider activities is exceptionally difficult, as adversaries operate under authorized roles and execute actions that mimic benign developer behaviors, such as downloading repositories, altering source code, or reading intellectual property files outside normal operational contexts.

To address the limitations of perimeter-centric security models, the Zero Trust Architecture (ZTA) framework, formalized by the National Institute of Standards and Technology (NIST) in Special Publication 800-207, has emerged as the leading paradigm for secure enterprise environments. Zero Trust operates under the core tenet of "never trust, always verify." It dictates that access to any resource must be continually authorized, authenticated, and validated based on the context of the requesting entity. In a Zero Trust environment, request evaluation is governed by the Policy Decision Point (PDP), which determines access privileges and instructs the Policy Enforcement Point (PEP) to allow, restrict, or deny interactions. A fundamental requirement of ZTA is the continuous evaluation of subject trust, where static credentials are replaced by dynamic, real-time trust scoring systems that integrate multiple signal streams, including user identity, device health, and behavioral telemetry.

Despite the conceptual clarity of NIST SP 800-207, current ZTA implementations suffer from significant limitations when applied to insider threat detection. Existing trust engines typically rely on static, rule-based policies or basic threshold checks that fail to capture the complex, evolving patterns of insider abuse. While machine learning (ML) models have been introduced to detect behavioral anomalies, these systems generally operate asynchronously as offline detection tools rather than inline components of the PDP. Consequently, access decisions remain detached from real-time behavioral risk assessments. Furthermore, typical ML models function as black boxes, providing classification outputs without explainable context. At the access decision point, security administrators and the PDP require transparent, explainable reasoning to justify access restriction or MFA challenges. Without inline Explainable AI (XAI) capabilities, security operators cannot verify why a developer was restricted, leading to administrative overhead, false-positive fatigue, and friction in legitimate software engineering workflows.

To overcome these challenges, this paper presents a Zero Trust-aligned framework that implements explainable insider threat detection and adaptive access control. The primary contributions of this work are as follows:
1) A two-stage hybrid machine learning pipeline combining an unsupervised Isolation Forest (IF) for anomaly score feature generation with a supervised Random Forest (RF) classifier to optimize detection under extreme class imbalance.
2) A formalized weighted Trust Score function that mathematically synthesizes multi-dimensional risk probabilities, anomaly scores, and identity confidence signals.
3) A rule-based policy engine mapping dynamic Trust Score bands to granular, automated policy actions, including authorization, multi-factor authentication challenge, restricted operations, and immediate administration alerts.
4) An inline explainability mechanism utilizing SHAP (SHapley Additive exPlanations) values to surface the top three features driving the risk assessment, presenting them as actionable reason codes at the access decision point.
5) An end-to-end containerized prototype integrating Keycloak as the PEP, FastAPI as the PDP, and a React-based administrative dashboard, validated using a log replay simulator running the CERT Insider Threat Dataset r4.2.
